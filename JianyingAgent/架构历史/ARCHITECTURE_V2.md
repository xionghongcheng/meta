# JianyingAgent V2 架构总览

版本日期：2026-05-28

## V2 定位

JianyingAgent V2 不再把“内容大脑”和“自动粗剪”硬写成一条大链路，而是升级为：

> 一个共享底座 + 一个内容大脑 Agent + 两条专业执行链路

V2 的核心目标：

> 把“账号理解”和“执行流程”拆开，让系统既能持续理解这个账号，也能分别服务拍前创作和拍后粗剪。

## 为什么从 V1 升级到 V2

V1 已经证明两件事成立：

- 你的账号确实可以被结构化为“内容大脑”
- 自动粗剪确实可以作为独立能力往下做

但 V1 也暴露了一个边界问题：

- `选题写稿 Agent` 解决的是拍前问题：拍什么、为什么拍、怎么讲
- `自动粗剪 Agent` 解决的是拍后问题：拍完后哪些素材该留、怎么先拼起来

把两者直接写在一个大 Agent 里，会导致：

- 创作阶段和执行阶段混在一起
- 编导判断逻辑和媒体处理逻辑耦合
- “内容大脑”既像知识库又像执行器，概念不清
- 后面很难继续扩展拍摄现场、复盘、商业化等能力

所以 V2 的调整方向不是否定 V1，而是把 V1 里混在一起的概念拆清楚。

## V2 的核心拆分

V2 把原来笼统的“内容大脑”拆成两部分：

- `共享底座`
  负责存放长期知识和记忆
- `内容大脑 Agent`
  负责读取共享底座，理解当前任务，做上游判断

所以不是：

```text
共享底座 = 内容大脑
```

而是：

```text
共享底座 = 原内容大脑里的静态部分
内容大脑 Agent = 原内容大脑里的动态判断部分
```

可以把它理解成：

- 共享底座是长期记忆
- 内容大脑 Agent 是会思考的脑子

## V2 总体架构

V2 采用：

```text
用户输入
  ↓
内容大脑 Agent
  ↓
调度层 / Router
  ├─ 拍前任务 → 选题写稿 Agent
  └─ 拍后任务 → 自动粗剪 Agent
```

同时，三类 Agent 共同读取同一个共享底座：

```text
共享底座
  ├─ 账号世界观
  ├─ 人物设定
  ├─ 内容主线
  ├─ 关系细节
  ├─ 场景规则
  ├─ 商业化规则
  └─ 历史故事库
        ↑
        ├─ 内容大脑 Agent
        ├─ 选题写稿 Agent
        └─ 自动粗剪 Agent
```

这里的关键不是“谁独占内容大脑”，而是：

> 共享底座是公共上下文，内容大脑 Agent 负责先理解任务，再把任务分发给专业执行 Agent。

## V2 四层结构

### 1. 共享底座层

负责沉淀长期知识，不直接执行任务。

包含：

- 账号世界观
- 人物设定
- 长期故事主线
- 关系细节
- 场景规则
- 商业化规则
- 历史故事库
- 文风模板

它回答的不是“这次怎么做”，而是“这个账号一直是什么样”。

### 2. Agent 层

负责理解任务和产出可执行结果。

包含三个核心 Agent：

- `内容大脑 Agent`
- `选题写稿 Agent`
- `自动粗剪 Agent`

三者都可以调用共享底座，但职责不同。

### 3. 调度层

负责接任务、识别任务类型、组织上下文、分发调用。

它不直接写脚本，也不直接剪视频。

它只做三件事：

- 判断任务属于拍前还是拍后
- 给目标 Agent 组装上下文
- 接收结果并返回标准输出

### 4. Pipeline 层

负责偏执行型、流程型能力。

尤其是自动粗剪链路里的：

- 素材扫描
- 音频抽取
- Whisper 转写
- 片段筛选
- 时间线拼接
- 剪映工程导出

这层更像“可复用的工序”，不应该和编导判断逻辑混在一起。

## 三个核心 Agent 的边界

### 内容大脑 Agent

这是 V2 的总控理解层，不是普通执行 Agent。

它负责：

- 读取共享底座
- 识别当前任务是拍前还是拍后
- 判断本次任务属于哪条内容主线
- 判断四个人物谁应做主角
- 判断是否撞题、是否适合商单
- 生成标准化任务单

它更像“总编导 Agent”或“上下文决策 Agent”。

### 选题写稿 Agent

这是拍前链路的核心 Agent。

它负责：

- 判断这条值不值得拍
- 明确这件事在你家为什么不普通
- 生成选题方向
- 生成脚本初稿
- 生成拍摄清单
- 生成必留原声点
- 对脚本做独立 review

它调用共享底座，是为了“写对”：

- 人物不要写串
- 重复题要找到新角度
- 生活细节要真实
- 商业化不能压过真实问题

### 自动粗剪 Agent

这是拍后链路的核心 Agent。

它负责：

- 读取素材和任务单
- 识别可用片段
- 优先保留关键原声和高光细节
- 规划时间线
- 导出粗剪结果和剪映工程

它调用共享底座，是为了“剪对”：

- 知道哪类人物反应更值得保留
- 知道结尾应该收在什么情绪
- 知道哪些生活颗粒对这个账号比炫技更重要

## V2 的连接方式：标准化任务单

V2 不建议让 Agent 之间直接传一堆自然语言描述，而是通过标准化对象连接。

建议最少定义四类对象：

- `topic_brief`
  这条题属于哪条主线、主角是谁、核心矛盾是什么

- `script_package`
  开头钩子、正文结构、拍摄清单、必留原声、结尾落点

- `roughcut_job`
  本次粗剪优先保留谁、哪些原声不能丢、节奏怎么收

- `roughcut_result`
  片段清单、粗排时间线、导出位置、剪映工程路径

这层的意义是：

- 降低 Agent 之间的耦合
- 方便后续换模型、换提示词、换执行流程
- 让系统边界清楚，不靠“约定俗成”传信息

## V2 与当前项目的映射关系

V2 不是推倒重来，而是对当前目录做重新归位。

### 当前内容大脑资产

这些在二阶段已经归入 `shared_base`：

- `shared_base/canon/*.md`
- `shared_base/memory/stories.json`

### 当前故事记忆能力

这些应拆到 `shared_base/memory/`：

- `core/content_memory.py`

可进一步拆成：

- `story_index.py`
- `ingest.py`

### 当前选题与脚本能力

这些在当前实现中已经归入 `agents/topic_writer_agent/`：

- `content_assistant.py`
- `core/script_assistant.py`

### 当前自动粗剪能力

这些在当前实现中已经归入 `pipelines/roughcut_pipeline/`：

- `skills/skill_1_extractor.py`
- `skills/skill_2_transcriber.py`
- `skills/skill_3_analyzer.py`
- `skills/skill_4_editor.py`
- `skills/skill_5_exporter.py`

### 当前总入口

`core/agent.py` 不再承担“所有东西都塞进去”的角色。

V2 中，它应被拆开，改为：

- 拍前由 `内容大脑 Agent + 选题写稿 Agent` 接管
- 拍后由 `内容大脑 Agent + 自动粗剪 Agent` 接管

## V2 的核心原则

- 知识、判断、执行三件事必须分开
- 共享底座是公共上下文，不归某一个 Agent 私有
- `内容大脑 Agent` 是总控理解层，不和执行 Agent 抢活
- `选题写稿 Agent` 只解决拍前问题
- `自动粗剪 Agent` 只解决拍后问题
- 拍前与拍后通过标准化任务单连接
- 不追求一个万能 Agent 包办一切

## V2 的工作流

### 拍前链路

```text
用户输入选题意图
  ↓
内容大脑 Agent
  ↓
判断主线 / 主角 / 风险 / 是否撞题
  ↓
选题写稿 Agent
  ↓
选题结论 / 脚本 / 拍摄清单 / 必留原声点
```

### 拍后链路

```text
用户输入素材和剪辑目标
  ↓
内容大脑 Agent
  ↓
补齐本期任务单
  ↓
自动粗剪 Agent
  ↓
转写 / 片段筛选 / 时间线粗排 / 剪映工程导出
```

## V2 的落地顺序

为了避免一次性改太多，建议按下面顺序推进：

### 第一步

已完成：新增目录和命名，并保留兼容入口。

### 第二步

已完成：`shared_base` 独立出来，内容资产和故事库从执行逻辑中分离。

### 第三步

已完成：拆出 `内容大脑 Agent` 和 `选题写稿 Agent`，让拍前链路独立成块。

### 第四步

已完成：粗剪能力迁入 `roughcut_pipeline/`，并保留 `skills/` 兼容包装层。

### 第五步

已完成：补上 `contracts/`，统一 Agent 间的输入输出结构。

### 第六步

已完成第一版：CLI 和调度层已经进入新的 V2 路由；后续只需要继续收缩兼容层。

### 第七步

已完成：历史示例脚本与 smoke test 已分别归到 `examples/` 和 `tests/smoke/`，根目录只保留兼容跳转。

## V2 不做什么

- 不把所有能力继续塞回一个总 Agent
- 不让“内容脑”再次同时承担知识库和执行器两种身份
- 不让拍前判断和拍后执行共用混乱边界
- 不急着追求一步生成最终成片
- 不把共享底座做成某个 Agent 的私有依赖

## V2 的一句话总结

> V1 是“统一内容大脑串起全链路”，V2 则是“共享底座提供长期记忆，内容大脑 Agent 负责上游判断，选题写稿和自动粗剪分别走自己的专业链路”。

这次升级的核心，不是多做几个目录，而是把：

- 什么是长期知识
- 什么是上游判断
- 什么是执行流程

这三件事真正拆开。
